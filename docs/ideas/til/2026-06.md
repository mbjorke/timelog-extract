# TIL — 2026-06

## 2026-06-12

- **Electron desktop apps share a de-facto local-evidence stack — worth standardizing one reader.** Claude Desktop and Lovable Desktop both: (1) keep their real activity server-side, (2) leave it only in the **Chromium "simple cache"** (`Cache/Cache_Data/*_0`: 20-byte header → key → response stream → EOF record), (3) compress bodies with modern HTTP codecs — **Lovable=brotli, Claude=zstd** (Cloudflare/edge defaults). As more tools ship on Electron this pattern will recur, so the Claude/Lovable collectors should lean on **one shared `chromium_cache` reader** (parse simple-cache entry + decode brotli/zstd) instead of re-implementing per app. The per-app part is only: which URL path to look for, and which fields to read.
- **Cache-reading is a last-resort source, not a default.** Only apply it to Electron apps whose primary data is server-side (Claude Desktop, Lovable Desktop). IDE forks (Cursor/Windsurf/Antigravity/Codex) and CLIs (Claude Code/Gemini/Codex) write first-class structured local logs — never scrape their cache.
- **Codecs as optional extras, not hard deps.** `brotli`/`zstandard` are binary wheels; keep the lean base install and gate desktop-cache collectors behind `pip install timelog-extract[desktop-cache]` with graceful no-op + a `gittan doctor` hint when missing. zstd is stdlib only from Python 3.14; Gittan targets 3.9+, so the third-party `zstandard` is required for now.
- **Net-new hours vs evidence quality are different wins.** Claude Code on the validation day was ~2.6h gross (morning), but observed time was already at/above the Screen Time ceiling (≈12.9h vs 12.6h) because Chrome covered the same window — so net-new daily total ≈ 0. The collector's real value that day was upgrading passive Chrome context to `direct_work_evidence` and firming per-project attribution. It *would* be net-new hours on a Claude-only morning. Always check the Screen Time headroom before claiming a source "adds hours".
