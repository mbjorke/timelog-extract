## 2026-07-22

- **`MERGE_GATE: CLEAR` (0 unresolved threads) is not "reviewed" — it can mean "the bot has not run yet".** Refines the 2026-07-10 lesson (which was about *unresolved* threads): here the gap is *timing*. PR #430 was reported green on "CodeRabbit pass + 0 unresolved threads" and merged **17 seconds after** Qodo posted 4 comments (2 security). A reviewer that has not posted yet is indistinguishable from a clean review under a thread-count check. Before merging, confirm **both** CodeRabbit *and* Qodo have a *completed review on the latest commit SHA*; treat a silent bot as un-reviewed (BLOCKED), never as clean. Rate-limited CodeRabbit is the common trigger — it posts nothing, so the window can close on silence. Structural fix tracked in #326 (require a positive review verdict, not merely absent threads); manual merges bypass the rabbit-loop gate entirely and rely wholly on this discipline.

## 2026-07-02

- **Two agents in one clone will collide silently — recover via reflog + a safety tag.** Claude Code and Cursor both worked `task/*` branches in the same working tree; a reused branch name + a checkout/reset left one agent's committed work dangling (no branch pointed at it). It was recovered from `git reflog` and pinned with `git tag safety/<name> <sha>`. Guardrails for next time: give each lane a **distinct, clearly-owned branch name**, work in your **own `git worktree`**, `git tag safety/...` before stepping away, and check local lanes/worktrees **before** branching (what `scripts/rabbit_workflow_context.sh` now automates — GH #240).
- **A fresh worktree is not an isolated dev environment.** `scripts/run_lint.sh` (run by `run_autotests.sh`) looks for `<root>/.venv/bin/ruff`; a new worktree has no `.venv`, so the lint gate exits non-zero under `set -e` and the unit tests never run — which misreads as a **test failure**. Fix: symlink the main clone's env (`ln -sfn <main-clone>/.venv .venv`) and add `.venv` to `.git/info/exclude`, or `pip install -e '.[dev]'` in the worktree.
- **Shared git identity means commits aren't agent-attributable.** Every agent commits as the same `user.name`/`user.email`, so history (and reflog) shows *what* changed, not *who* did it.
