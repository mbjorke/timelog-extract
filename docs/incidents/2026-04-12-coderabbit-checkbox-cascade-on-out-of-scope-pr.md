# Incident: CodeRabbit checkbox cascade on out-of-scope PR

**Date:** 2026-04-12  
**Branch:** `claude/gittan-launch-review-4a5DD` → PR #14  
**Status:** Resolved (work landed, but scope was lost)

## Summary

A review branch already carrying an out-of-scope security fix received a CodeRabbit
review. The maintainer clicked "Add unit tests" in the CodeRabbit chat interface.
This single checkbox triggered a cascade of six further commits — each one technically
correct, each one pulling the branch further from its original scope of "review
document."

## Commit sequence

| # | Author | Commit | Triggered by |
|---|---|---|---|
| 1 | Claude | `fix(security): use parameterized queries` + review doc | Original brief (out-of-scope fix already bundled) |
| 2 | Claude | `fix(security): escape LIKE wildcards` | CodeRabbit review suggestion relayed in session |
| 3 | CodeRabbit | `📝 CodeRabbit Chat: Add unit tests` | **Maintainer clicked "Add unit tests"** ← inflection point |
| 4 | CodeRabbit bot | `fix: apply CodeRabbit auto-fixes (#15)` | Maintainer clicked auto-fix checkbox |
| 5 | Cursor | `test: split Chrome tests` (500-line CI failure) | CI failure caused by commit 3 |
| 6 | Cursor | `test(chrome): assert real SQL matches` | CodeRabbit review of commit 5 |
| 7 | Cursor | `docs: note planned PyPI release` | Scope drift; Cursor adding context |

The review document — the original deliverable — is commit 1. Everything after it
was generated by reviewing the out-of-scope security fix, not the review document.

## Root cause

The maintainer approved a CodeRabbit suggestion ("Add unit tests") on a PR whose
scope was already compromised. CodeRabbit reviewed the code as presented — it had no
awareness that the security fix was out of scope. Its suggestion was technically
correct. The checkbox click was the decision point.

Once tests existed, CI enforced the 500-line policy against them, pulling Cursor into
the branch. Each agent that touched the branch did correct work in isolation; the
problem was that the branch scope had already been abandoned.

## The cascade pattern

```
agent overreaches (out-of-scope fix)
    ↓
CodeRabbit reviews the overreach (correctly)
    ↓
maintainer clicks suggestion on out-of-scope work  ← decision point
    ↓
new work (tests) triggers CI failure
    ↓
another agent fixes CI failure
    ↓
CodeRabbit reviews that fix
    ↓
...
```

Each step is locally reasonable. The cascade is only visible when you look at the
whole sequence.

## What the right move was

At commit 1 — or at the latest when CodeRabbit first posted suggestions — the correct
response was:

> "This branch has a review document and an unrelated security fix. I'm going to split
> the fix into its own branch. Don't apply any suggestions here until the scope is
> clean."

## Impact

- Review document never formally evaluated; buried under six subsequent commits.
- PR #14 became a security fix PR with a review document attached, not a review PR.
- Three agents (Claude, CodeRabbit, Cursor) did legitimate work on illegitimate scope.

## Corrective actions

- `AGENTS.md` — Do not #1 and #1b added.
- `docs/til/2026-04-11-dont-fix-unrelated-things-in-an-rc.md` — covers agent side.
- `docs/til/2026-04-12-coderabbit-checkbox-as-scope-decision.md` — covers this
  maintainer-side pattern.
